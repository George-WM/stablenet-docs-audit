## D-015: EVM Compatibility 페이지 위치 미결정 (2026-04-21)
- **결정**: EVM Compatibility 페이지를 Reference 섹션에 유지할지, Architecture 섹션으로 이동할지 미결정
- **이유**: Reference는 개발자가 빠르게 찾기 좋고, Architecture는 내용의 성격상 더 적합할 수 있음. IA 재구성 후 최적 위치 재평가 필요
- **상태**: 미결정 — 추후 논의

## D-014: Week 1-2 P1 완료, Week 3-4 진행 (2026-04-10)
- **결정**: P1.1~P1.4 Quick Wins 완료 후 Week 3-4(P2) 진행
- **이유**: v0.2 프리뷰 확인 완료, 4개 PR 모두 merged, 기본 온보딩 구조 확립
- **결과**: v0.2에 Quick Start, Network Info, Changelog, API Reference, Quick Start Hub 5개 페이지 추가

## D-013: PR은 검수(review) 완료 후 올리기 (2026-04-10)
- **결정**: write → fact-check → review → PR 순서 확립
- **이유**: P1.1에서 팩트체크 전에 PR을 올렸더니, 수정 후 force push가 필요했음. 검수까지 끝난 뒤 PR이 깔끔함
- **교훈**: 파이프라인 순서를 지키면 재작업이 줄어든다

## D-012: 팩트체크 에이전트가 hex 변환 오류를 잡음 (2026-04-09)
- **결정**: 팩트체크 에이전트를 별도 세션으로 분리한 것이 유효했음
- **이유**: 수정 에이전트가 `0x206B`(=8299)를 8283의 hex로 잘못 작성. 팩트체크 에이전트가 v0.1 원본과 수학적 검증으로 `0x205B`가 맞다고 잡아냄
- **교훈**: 같은 세션에서 자기가 쓴 걸 자기가 검수하면 이런 오류를 놓친다. 별도 세션이 필수

## D-011: 검수 에이전트가 ko 코드 번역 문제 잡음 (2026-04-09)
- **결정**: 검수 에이전트의 "Bilingual Consistency" 체크가 유효
- **이유**: P1.3에서 수정 에이전트가 ko 코드 블록의 플레이스홀더를 한글로 번역함 (`"0xYourAddress"` → `"0x계정주소"`). 검수 에이전트가 FAIL 처리
- **교훈**: CLAUDE.md에 "코드는 번역하지 않음" 규칙이 있어도 에이전트가 놓칠 수 있다. 별도 검수가 안전장치

## D-010: v0.2는 완전히 새로운 docs로 작성 (2026-04-09)
- **결정**: v0.1 스타일 무시, v0.2는 독립된 개발자 온보딩 문서로 작성
- **이유**: v0.1은 프로토콜 엔지니어/노드 운영자 대상의 깊은 기술 문서. Primary ICP(EVM 개발자)에게는 부적합. 스타일을 맞추려다 어중간해지는 것보다 완전히 새로 쓰는 게 낫다
- **원칙**: 기술 정보는 v0.1에서만 가져오되(창작 금지), 표현은 완전히 바꾼다

## D-009: 3개 에이전트 별도 세션 (수정/팩트체크/검수) (2026-04-09)
- **결정**: 수정/팩트체크/검수를 각각 다른 Claude Code 세션에서 실행
- **이유**: 같은 세션이면 자기가 쓴 걸 자기가 검수하는 한계. 별도 세션은 "fresh eyes" 효과
- **트레이드오프**: 세션 전환 번거로움 vs 품질 보장. 실제로 D-012, D-011에서 별도 세션이 오류를 잡았으므로 정당화됨

## D-008: docs-pipeline.sh 오케스트레이터 (2026-04-09)
- **결정**: 완전 자동화가 아닌 단계별 수동 트리거 방식
- **이유**: 수정 → 상윤 확인 → 팩트체크 → 상윤 확인 → 검수 흐름에서 사람의 판단이 매 단계 필요. 완전 자동화하면 FAIL인데 다음 단계로 넘어가는 낭비 발생
- **구현**: `./docs-pipeline.sh write/fact-check/review P1.N` + status 추적

## D-007: 평가 스키마 재설계 — ICP 기반 4축 (2026-04-02)
- **결정**: 일반 블록체인 체크리스트 → ICP 의사결정 여정 기반 4축으로 재설계
- **이유**: 경쟁사 8곳의 성격이 전부 다름(체인, 인프라, DeFi, 컴플라이언스). 동일 항목으로 채점하면 사과와 오렌지를 비교. ICP가 docs를 보는 관점으로 평가해야 의미 있음
- **4축**: Developer Onboarding(0.35), Institutional Confidence(0.15), Stablecoin Depth(0.25), Docs Quality(0.25)

## D-006: Axis 2 가중치 0.15로 낮춤 (2026-04-02)
- **결정**: Institutional Confidence 가중치를 다른 축 대비 낮게 설정
- **이유**: KRW 스테이블코인 규제가 미확정. 컴플라이언스 docs를 쓸 수 없는 상태에서 이 축을 높게 잡으면 "못 고치는 Gap"이 최우선으로 나옴
- **향후**: 규제 확정 시 가중치 상향 + 별도 프로젝트로 분리

## D-005: 컴플라이언스 docs 스코프 제외 (2026-04-02)

## D-004: Stable.xyz 경쟁사 추가, Tether 제거 (2026-04-02)
- **결정**: 경쟁사 9곳 → Tether 제거 후 8곳, Stable.xyz 추가
- **이유**: Stable.xyz는 USDT 네이티브 L1 "stablechain"으로 StableNet과 가장 직접적인 포지셔닝 경쟁. Tether는 발행사이지 체인이 아니라서 Developer Docs 비교 가치 낮음 + docs 접근 불가
- **교훈**: preflight에서 URL 접근성 사전 확인이 유효 (Tether, Maroo URL 변경 발견)

## D-003: 순차 실행 (병렬 아님) (2026-04-02)
- **결정**: 경쟁사 분석을 순차 실행, 병렬 아님
- **이유**: 비용은 동일(총 토큰 수 같음). 디버깅 용이성이 핵심 — 첫 1-2개 결과를 보고 프롬프트/스키마 보정 가능. 멀티 에이전트 첫 경험이므로 복잡도 최소화
- **결과**: 실제로 Paxos에서 JSON 파싱 에러 발생 → 순차 실행이라 즉시 발견/수정 가능했음

## D-002: Maroo 파일럿 후 스키마 보정 (2026-04-02)
- **결정**: orchestrator.sh에 Maroo 파일럿 후 명시적 pause 삽입
- **이유**: 8곳 전부 돌린 후 스키마가 잘못됐으면 전부 재실행. Maroo 1곳만 먼저 돌려서 스키마/프롬프트 검증 후 나머지 진행

## D-001: Claude Code 기반 멀티 에이전트 (Cowork 아님) (2026-04-02)
- **결정**: Claude Code + orchestrator.sh 기반 멀티 에이전트 파이프라인
- **이유**: 핵심 작업이 웹 크롤링 → 구조 분석 → 비교 → 리포트 생성 파이프라인. Claude Code가 GitHub repo 직접 접근, 스크립트 실행, 대량 텍스트 처리 모두 가능. Cowork는 로컬 파일 관리에 강하지만 이 작업에는 부적합
- **하네스**: 동일 평가 스키마(YAML)로 출력 강제 → Phase 3에서 합성 가능

## 메타 원칙 (프로젝트 전반에 걸쳐 반복된 패턴)

1. **경쟁사 설정은 내부 판단을 우선** — 초기 벤치마크 대상(일반 L2)은 실제 경쟁 구도와 맞지 않았음. 도메인 전문가(상윤)의 직접 수정이 필요
2. **사전 검증이 재작업을 줄인다** — preflight(URL 체크), 파일럿(Maroo), 팩트체크(별도 세션) 모두 "미리 잡는" 구조
3. **구조화된 출력이 합성을 가능하게 한다** — 하네스 엔지니어링의 핵심. 자유 형식 에세이가 아니라 스키마 강제
4. **"못 하는 것"을 빨리 인정하면 "할 수 있는 것"에 집중할 수 있다** — 컴플라이언스 docs 스코프 제외가 대표적 사례
